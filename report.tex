\section{Obiettivo del Progetto}

Creeping Crawler è una pipeline end-to-end per l'acquisizione e l'analisi di contenuti testuali da sorgenti web eterogenee, progettata per alimentare sistemi RAG e modelli linguistici. Il sistema scarica pagine web tramite Crawl4AI, le trasforma in testo strutturato tramite parser specifici per dominio, e ne valuta la qualità di estrazione confrontando l'output con un Gold Standard costruito manualmente.

% ─── 2. Organizzazione del codice ────────────────────────────────────────────

\section{Organizzazione del Codice}

Il progetto è suddiviso in due container Docker orchestrati da \texttt{docker-compose.yaml}: \textbf{backend} (FastAPI, porta 8003) e \textbf{frontend} (FastAPI + Jinja2, porta 8004).

\paragraph{Backend}

\begin{itemize}
  \item \textbf{\texttt{src/lib/crawling/}} — Modulo di acquisizione basato su Crawl4AI. \texttt{crawler.py} espone tre funzioni pubbliche: \texttt{fetch\_page} (crawl live), \texttt{fetch\_page\_from\_html} (parsing da HTML già disponibile via prefisso \texttt{raw:}), e \texttt{fetch\_page\_for\_url} (usa l'HTML salvato nel Gold Standard come sorgente primaria; ricade sul crawl live solo se non disponibile). Questa scelta è motivata dalla natura dinamica dei domini assegnati: siti come XE aggiornano frequentemente i dati sulle valute monetarie, per cui ricrawlare la pagina live avrebbe introdotto uno sfasamento rispetto al \texttt{gold\_text} costruito manualmente, inquinando la valutazione. La sottocartella \texttt{domains/} contiene un \texttt{CrawlerRunConfig} dedicato per ciascun dominio, con esclusione di tag, selettori CSS e form non pertinenti; \texttt{CacheMode.BYPASS} garantisce contenuto sempre aggiornato.

  \item \textbf{\texttt{src/lib/parsers/}} — Interfaccia astratta \texttt{ContentParser} con implementazioni specifiche in \texttt{domains/} (Wikipedia, ESPN, CNBC, XE). Ogni parser opera sul Markdown prodotto da Crawl4AI: interrompe la raccolta alle sezioni boilerplate (es. ``Note'', ``Related Content''), rimuove pattern residui tramite espressioni regolari (note a piè di pagina, link di navigazione, didascalie), e restituisce testo pulito in formato Markdown.

  \item \textbf{\texttt{src/lib/evaluation/}} — Pipeline di valutazione in tre moduli: \texttt{tokens.py} (stripping Markdown via mistune/BeautifulSoup, normalizzazione Unicode), \texttt{token\_level.py} (metriche set-based: precision, recall, F1), \texttt{similarity.py} (metriche vettoriali: cosine, Jaccard, excess ratio).

  \item \textbf{\texttt{src/lib/gold\_standard/}} — Accesso a \texttt{domains.json} e ai file JSON in \texttt{gs\_data/}, uno per dominio, ciascuno con le entry rappresentative contenenti \texttt{url}, \texttt{domain}, \texttt{title}, \texttt{html\_text} e \texttt{gold\_text}.

  \item \textbf{\texttt{src/routes/}} — Quattro router FastAPI (\texttt{domains}, \texttt{parse}, \texttt{gold}, \texttt{evaluate}), ciascuno con schemi Pydantic dedicati in \texttt{src/schemas/}.
\end{itemize}

\paragraph{Frontend}

\begin{itemize}

Server FastAPI con rendering Jinja2. La route \texttt{/compare} recupera il parsing dal backend, calcola le metriche se l'URL è nel Gold Standard, e visualizza quattro pannelli testuali (HTML grezzo, Markdown parsato, testo pulito, Gold Standard) più un diff interattivo.

\end{itemize}

% ─── 3. Valutazione Parser ───────────────────────────────────────────────────

\section{Valutazione dei Parser}

\paragraph{Metodologia.} Ogni parser viene valutato su 5 URL per dominio tramite \texttt{GET /full\_gs\_eval}. Il testo estratto e il Gold Standard vengono entrambi passati per \texttt{strip\_markdown} prima del confronto, in modo da normalizzare quote tipografiche, trattini Unicode ed entità HTML. Le metriche set-based operano su insiemi di token unici (case-sensitive, separazione per spazio); le metriche vettoriali (cosine, Jaccard, excess ratio) operano sui conteggi di frequenza tramite \texttt{Counter}.

\paragraph{Considerazioni qualitative.} La configurazione di Crawl4AI tramite \texttt{CrawlerRunConfig} (esclusione di selettori CSS specifici per dominio) è il primo livello di filtro. Il parser Python agisce sul Markdown residuo: per Wikipedia, le sezioni di servizio (``Note'', ``Bibliografia'', ``Voci correlate'') vengono interrotte con precisione; per ESPN e CNBC i pattern regex rimuovono promozioni video, link isolati e didascalie; per XE il contenuto strutturato è già quasi privo di rumore dopo la configurazione. Il confronto visuale tramite Monaco Diff Editor nel frontend ha guidato il tuning iterativo dei pattern di esclusione.

% ─── 4. Bonus/Extra ──────────────────────────────────────────────────────────

\section*{Bonus/Extra}

\paragraph{Similarity Eval (metriche aggiuntive).} Oltre alla \texttt{token\_level\_eval} obbligatoria, è stato implementato un secondo gruppo di metriche basato su vettori di frequenza: \emph{cosine similarity} (misura la distribuzione dei token), \emph{Jaccard similarity} (penalizza sia token extra che mancanti), ed \emph{excess ratio} (frazione dei token estratti non coperti dal Gold Standard, più basso è meglio). Questi valori completano la valutazione set-based rilevando testo ridondante non catturato da precision/recall.

\paragraph{Script \texttt{crawl\_gs.py} e \texttt{Makefile}.} Uno script autonomo scarica tutti gli URL del Gold Standard e salva l'output strutturato in \texttt{gs\_results/} in cinque formati: HTML grezzo e ripulito, Markdown, testo stripped e testo parsato. L'opzione \texttt{--update-json} aggiorna automaticamente il campo \texttt{html\_text} nei file JSON del Gold Standard per permettere il re-parsing offline senza ricrawlare. Il Makefile automatizza l'intero ciclo di sviluppo: creazione degli ambienti conda, installazione dipendenze (incluso Playwright/Chromium), avvio dei server, crawling, snapshot dei requirements e pulizia degli ambienti.

\paragraph{Monaco Diff Editor.} La pagina dei risultati del frontend integra Monaco Editor (la stessa base di VS Code) in modalità diff affiancato. Quando l'URL è nel Gold Standard, il pannello mostra token per token la differenza tra il testo pulito estratto dal parser e il Gold Standard, evidenziando esattamente cosa è stato aggiunto o mancato. Questo ha reso il tuning dei parser molto più rapido rispetto al confronto manuale su file di testo.

\paragraph{Endpoint aggiuntivi.} Oltre agli endpoint obbligatori è stato implementato \texttt{GET /gold\_standard\_urls}, che restituisce la lista completa degli URL nel Gold Standard ed è usato dal frontend per popolare il menù a tendina e determinare se un URL inserito manualmente ammette valutazione.
